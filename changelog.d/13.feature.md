## CLI-Side Fetching for Skill Sources the Server Can't Reach

`dot-ai skills generate` can now read a skill source on the CLI host and upload it for the server to render, for sources the server itself cannot reach or authenticate. Previously every source was fetched server-side via `--repo`, which assumes the server can both route to and authenticate against the repo — unworkable for sources behind SSO/OIDC/device-attested VPNs, hardened clusters with no egress and no static-token path, or work-in-progress skills that live only on a developer's disk.

Two additive, mutually-exclusive source flags fill that gap; both still render server-side, so argument-taking skills behave exactly as with `--repo`:

- **`--repo-fetch <git-url>`** `[--repo-path <subdir>] [--repo-branch <branch>]` — the CLI clones the repo using the *host's* local git stack (SSH agent, `git credential`, `~/.gitconfig`, `GIT_SSH_COMMAND`, …), then uploads the result. The skill's `source:` frontmatter is the git URL with any credentials scrubbed. Authentication is the host git stack, not `DOT_AI_GIT_TOKEN`.
- **`--repo-dir <path> --source-label <label>`** — reads skills from a local directory with no network or clone and uploads them, tagging `source: local:<label>` (auto-prefixed with your user/host for cross-machine uniqueness). It is off by default and opt-in: set `DOT_AI_ALLOW_REPO_DIR=1`; paths under `/tmp` or world-writable directories are refused, with an optional base-path allowlist via `DOT_AI_REPO_DIR_ALLOW`.

Supporting capabilities: `--repo-fetch` keeps an XDG clone cache (`~/.cache/dot-ai-cli/repos/`) that incrementally fetches on re-runs and serializes concurrent runs of the same URL, with `--no-cache` for a throwaway clone and `dot-ai skills cache prune --older-than <duration>` to garbage-collect it; unchanged sources skip re-upload via a content hash and are transparently re-uploaded if the server has evicted them. `--install-hook` round-trips the new flags (credentialed URLs are scrubbed before being written to `settings.json`, and a `--repo-dir` hook reads `DOT_AI_ALLOW_REPO_DIR` from the environment at run time rather than baking it in). The existing source-tagging composes cleanly across env-var, `--repo`, `--repo-fetch`, and `--repo-dir` sources, with first-source-wins on name collisions. No credential or credentialed URL appears in logs, output, generated frontmatter, or the cache.

Prefer `--repo` + `DOT_AI_GIT_TOKEN` (server-side fetch) whenever a static credential can reach the source — it keeps credential audit centralized. Use `--repo-fetch`/`--repo-dir` only as the escape hatch for sources the server fundamentally cannot reach or authenticate.

This feature requires the companion server-side ingestion endpoint and ships in the same release. Backward compatibility is preserved: with none of the new flags, `dot-ai skills generate` behaves exactly as before.
